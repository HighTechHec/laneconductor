# Spec: Track 1062 — Deployment Context & lc deploy Command

## Problem Statement

LaneConductor gives AI agents full context about what the code does (`product.md`, `tech-stack.md`) and how it's developed (`workflow.md`), but has no concept of **how it gets deployed**. This means:
- AI agents generate code without knowing the target runtime (Cloud Run vs Lambda vs Vercel)
- No standard way to trigger deployments from the CLI
- Deployment scripts are ad-hoc per-project (deploy.sh, Makefiles)
- Secrets and credentials end up hardcoded or in .env files that get committed
- No AI-guided deployment setup — users figure it out from scratch every time

## Core Concept

Deployment becomes a first-class citizen alongside code development:

```
conductor/
├── tech-stack.md        ← what it's built with (existing)
├── deployment-stack.md  ← how and where it deploys (NEW)
└── tracks/              ← development work (existing)
```

`deployment-stack.md` is a **context file** — read by AI agents during `implement` and `review` so they can generate deployment-aware code, Terraform, CI config, and environment variable handling automatically.

## Requirements

### REQ-1: `lc setup-deploy` — AI-Guided Deployment Setup

Claude-driven wizard (same pattern as `lc setup scaffold`):
1. Scans existing project for deploy signals (`deploy.sh`, `Makefile deploy` target, `Dockerfile`, `firebase.json`, `vercel.json`, `.github/workflows/`)
2. Asks targeted questions to understand the deployment topology
3. **Component-based selection** — user picks independently per layer (not monolithic templates):
   - **Frontend**: Firebase Hosting / Vercel / GCP Cloud Storage / S3+CloudFront / none
   - **Backend**: GCP Cloud Run / AWS Lambda / Vercel Functions / Firebase Functions / none
   - **Database**: Cloud SQL / Supabase / RDS / PlanetScale / none
   - **Secrets**: GCP Secret Manager / AWS Secrets Manager / Azure Key Vault / Vercel env vars / Supabase Vault
4. Verifies existing credentials for each selected component (see REQ-5)
5. If existing deploy scripts found: wraps them in `deploy.json`; if new project: generates deploy commands
6. Writes `conductor/deployment-stack.md` documenting the full topology
7. Writes `conductor/deploy.json` (machine-readable config for `lc deploy`)
8. Generates `.env.example` with all required env var names for CI migration
9. Updates `.gitignore` — **never writes secret values to any file**

### REQ-2: Named Deployment Presets (Not Templates)

Presets are **shorthand names** that pre-fill component selections — not files. User can accept or customize any component after selecting a preset.

| Preset name | Frontend | Backend | DB | Secrets |
|-------------|----------|---------|-----|---------|
| `firebase-full` | Firebase Hosting | Firebase Functions | Firestore | Firebase/ADC |
| `gcp-cloud-run` | — | Cloud Run | Cloud SQL (opt) | GCP Secret Manager |
| `gcp-full-stack` | Firebase Hosting | Cloud Run | Cloud SQL | GCP Secret Manager |
| `vercel` | Vercel | Vercel Functions | — | Vercel env vars |
| `aws-serverless` | S3+CloudFront | Lambda | RDS (opt) | AWS Secrets Manager |
| `supabase` | — | — | Supabase | Supabase Vault |

User can also describe a mix in plain language: *"Firebase hosting, Cloud Run backend, Supabase DB"* — Claude maps it to components.

### REQ-3: `lc deploy [env]` — Deploy Command

```bash
lc deploy              # deploy to default env (usually prod)
lc deploy prod         # explicit prod
lc deploy staging      # staging
lc deploy preview      # preview/PR deploy (Vercel, Firebase preview channels)
```

Reads `deploy.json` and executes the corresponding deploy command for the environment.
Shows progress, captures logs, updates track status if a track is active.

### REQ-4: `deployment-stack.md` — Context File

Human-readable markdown read by AI agents. Generated by `lc setup-deploy` and kept updated.

```markdown
# Deployment Stack

## Provider
GCP (Google Cloud Platform)

## Environments
- prod: europe-west1, project: makrodash
- staging: europe-west1, project: makrodash-staging

## Services
- Frontend: Firebase Hosting (makrodash.web.app)
- Backend: Cloud Run (macrodash-backend, europe-west1)
- Database: Cloud SQL (Postgres 15)
- Registry: Artifact Registry (europe-west1-docker.pkg.dev/makrodash/macrodash)

## Secret Management
GCP Application Default Credentials (ADC) for local dev.
Secret Manager for production secrets. Never in .env or code.

## IaC
Terraform in infra/. State in GCS bucket: makrodash-tf-state.

## Deploy Command
lc deploy prod  →  runs infra/deploy.sh prod
```

### REQ-5: Credential Verification + Zero-Secrets Policy

**Golden rule: no secret values ever touch the filesystem in a committed file.**

Claude verifies existing auth for each selected component by running the provider's identity check:

| Provider | Verification command | Documents as |
|----------|---------------------|--------------|
| GCP | `gcloud auth list` + `gcloud auth application-default print-access-token` | `auth: ADC (verified)` |
| AWS | `aws sts get-caller-identity` | `auth: IAM profile <name> (verified)` |
| Azure | `az account show` | `auth: Azure CLI (verified)` |
| Vercel | `vercel whoami` | `auth: Vercel CLI token (verified)` |
| Supabase | `supabase projects list` | `auth: Supabase CLI (verified)` |
| Firebase | `firebase projects:list` | `auth: Firebase CLI / ADC (verified)` |

On verification success: writes `auth: <method> (verified ✓)` to `deployment-stack.md`.
On failure: writes `auth: NOT CONFIGURED` + prints setup instructions (e.g. `gcloud auth application-default login`).

**`.env.example` generation**: Always generated with:
- All env var names required for CI (e.g. `GOOGLE_APPLICATION_CREDENTIALS`, `VERCEL_TOKEN`)
- Comments explaining each var: what it is, how to obtain it, ADC equivalent for local dev
- Never contains actual values

`lc setup-deploy` also enforces:
- `.gitignore` covers: `.env`, `*.tfvars`, `service-account*.json`, `*-key.json`, `.vercel`
- Warns if any of these patterns are currently tracked by git

### REQ-6: Local Build+Deploy Only (Phase 1)

`lc deploy` runs **locally** — the build and push happen on your machine. This fits LaneConductor's sovereign, local-first identity.

Remote CI/CD (GitHub Actions, Cloud Build, etc.) is **deferred to Phase 2**. The `deploy.json` schema reserves a `"ci"` key for future expansion:

```json
{
  "ci": null
}
```

When `ci` is set in Phase 2, `lc deploy --remote` will trigger the remote pipeline instead of running locally.

### REQ-6: `conductor/deployment-stack.md` added to scaffold

`/laneconductor setup scaffold` should offer to run `lc setup-deploy` after scaffolding, and if declined, create a stub `deployment-stack.md`:

```markdown
# Deployment Stack

## Status
Not configured. Run `lc setup-deploy` to set up deployment.
```

### REQ-7: AI Agent Awareness

When `deployment-stack.md` exists, agents automatically read it during:
- `implement` — so they write deployment-aware code (e.g. use Secret Manager SDK, not `process.env` directly)
- `review` — so they can flag code that leaks secrets or uses wrong env var patterns
- `quality-gate` — so they can verify infra files exist and secrets policy is followed

### REQ-8: `deploy.json` Machine-Readable Config

```json
{
  "template": "gcp-cloud-run",
  "environments": {
    "prod": {
      "command": "bash infra/deploy.sh prod",
      "region": "europe-west1",
      "project": "makrodash"
    },
    "staging": {
      "command": "bash infra/deploy.sh staging",
      "region": "europe-west1",
      "project": "makrodash-staging"
    }
  },
  "secrets": {
    "strategy": "adc+secret-manager",
    "keys": ["DB_PASSWORD", "JWT_SECRET", "STRIPE_KEY"]
  },
  "gitignore_additions": [".env", "*.tfvars", "service-account*.json"]
}
```

## Acceptance Criteria

- [ ] `lc setup-deploy` runs interactively and writes `deployment-stack.md` + `deploy.json`
- [ ] All 6 templates generate correct infra scaffolding
- [ ] `lc deploy [env]` reads `deploy.json` and runs the correct command
- [ ] No template ever writes a secret value to a file
- [ ] `.gitignore` is updated with secrets exclusions after setup
- [ ] `deployment-stack.md` stub created during scaffold if setup-deploy not run
- [ ] AI agents read `deployment-stack.md` during implement/review/quality-gate
- [ ] GCP ADC, AWS credential chain, Azure Managed Identity, Vercel CLI all documented in generated files
- [ ] `lc setup-deploy` detects existing deploy setup (deploy.sh, Makefile targets) and offers to migrate
